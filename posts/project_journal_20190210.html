Project Journal: February 10, 2019

I've ended up hitting a roadblock in the deployment process, as I can't get half of my QUnit tests to pass in the PhantomJS headless browser.

The issue at hand is peculiar. All of my tests which use the "Submit my answer" button are failing. The initial issue was that two useful lines of jQuery, e.preventDefault() and result false; were seemingly not preventing the form fields from appending the URL and being sent as a GET request.

I made the form explicitly POST (which is what it will eventually need), and that issue was resolved.

However, my headless browser experience different from the Chrome experience, where upon clicking "Submit my answer" the user would see a message appear in a span, based on their response (i.e. "Incorrect units", "Correct!", etc.)

I have a spent a couple days on this issue, partly because I was consistently getting more clarity on the issue the more I worked on it. However, and this is assuming I will not experience similar issues with another headless browser, moving to Puppeteer will at least provide me an opportunity to discover if my code is faulty in general or just for PhantomJS.

At this point, with one week left until my Six Week Versions are due, I am going to pursue using another headless browser, namely puppeteer, and save this PhantomJS puzzle as an issue for another day.

Running a process in parallel

The PhantomJS was not all for naught, as I learned how to configure my .travis.yml file in order to start the server prior to running the tests: 


language: node_js
node_js:
    - "stable"

cache:
    directories:
    - "node_modules"

before_script:
    - npm install connect
    - node src/server.js ../../ 8080 &
    - sleep 15

script: 
    - phantomjs src/test/run_qunit.js "http://localhost:8080/test"

The server is listening on port 8080 before the test script is run.
