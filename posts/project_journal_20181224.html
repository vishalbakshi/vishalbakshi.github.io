Project Journal: December 24, 2018

Entering the world of open source contributions

Today, I started my first attempt at open source contribution. The first-timers-only and similarly labelled issues on GitHub was what I was looking for. This repo has an awesome list of such open issues.

The first project I am attempting to contribute to is Public Lab's Image Sequencer. The particular issue which is flagged as first-timers-only is to create a test for one of their modules which converts an image file to QR code. The jsqr package has updated from its 0.2.2 version, and including that update without a test would leave them blind to anything that may break because of it.

I started by forking the repo to my GitHub account, cloning it onto my local computer, and practicing the git fetch upstream command. Initially as I was following the online GitHub documentation (which is great by the way) which showed the git fetch upstream master command, I ran into issues (i.e. that branch was not found) because the default branch on the upstream repo, which I had forked, was the main branch. The solution was, of course, that I had to fetch the main branch from upstream. Ta-da!

I then created a new branch, decodeqr-test, where I will start writing the tests.

Public Lab is using the tape package, which uses the "tap" philosophy, which stands for test anything. I was able to make a basic initial test run: 


    test('Initial test', function(t) {
        t.equal(true,true,'Initial test if true === true');
        t.end();
    })
which was a feat in itself, since it took me awhile to figure out that I had to npm run setup in order to install the dependencies in their project before I could require them!

My next tasks on this project are:

  • Understand what the DecodeQR module does
  • Understand the difference between "unit tests" and "functional tests"
  • Write one test for each unit
  • Write one test for each functionality

JS Podcasts

In order to fill my lack of intangible JavaScript and web development experience, I listened to my first set of programming related podcast episodes today. I have started listening to current and previous episodes of syntax.fm and have loved every minute.

Wes Bos and Scott Tolinski go through all sorts of topics related to the web development experience, from the breakdown of the State of JavaScript survey with one of its creators Sacha Grief, to best practices and tips for how to be well-liked (and/or how not to be disliked) at work.

In my opinion, the most insightful episodes as a beginner, have been the Hasty Treat - Where are they now? episodes, where Wes and Scott talk about technologies that were "hotness" back when, but have faded in popularity since. This bridges a vast lack of understanding I have of historical knowledge/origin stories for all things web development but especially JavaScript, with its breakneck progress into the future.

On starting projects

I have been going (two weeks now) to a fantastic meetup called OpenOakland which is held in City Hall most every Tuesday. One of the team members there mentioned that GitHub was a "learning wall" that you had to break through and not a learning curve that you could ease into. This ties into woes of starting on a project while holding the fear that I may not know enough to build it the "right way".

So I've jumped in and started creating projects that I've been chewing on for awhile. The first one, which will be the topic of many posts, I hope, is a web app I am creating for ET 130 (Applied Fluid Mechanics), one of the courses I teach at CCSF. There are a few more ideas that I have been needing to roll out, and I will hopefully do so this week.
