Project Journal: December 27, 2018

I experienced the implications of asynchronous functions the hard way today. I had written something like the following: 


        MongoClient.connect(

            // handle errors
            let db = client.db('database name')

            // call another function to create a new collection
            createNewCollection(db, function() { client.close() });

        );

        function createNewCollection(db, callback) {

            // create new collection in db
            db.createCollection('collection name', options);
            
            // closes the connection
            callback();
        
        }

        // function to be called later on in POST request handler
        function connectAndInsert(doc) {
            MongoClient.connect(
                // handle errors
                // insert new document into db
                client.close();
            );
        }

        app.route('/action-after-form')
           .post(function(req, res) {
               // create a new document from request body
               let newDocument = {key: val}
               connectAndInsert(newDocument);
               // return a response
           })

This, however, resulted in the following error message:

Error inserting document: MongoError: server instance pool was destroyed

Essentially, I was creating and closing the connection in two different locations in my code, asynchronously, and the timing on those events was not unfolding in the manner in which I was expecting, so the connection was closing before it inserted the new document.

One main takeaway, which is obvious in hindsight: don't expect anything about the timing of async functions.

Another main takeaway: I need to learn best practices for the MongoDB Node driver.

Documentation for the correct version

I fundamentally did not understand the API reference, leading to the issues above, and additionally, it turns out that the awesome reference I followed yesterday, MongoClient or how to connect in a new and better way, is quite outdated.

What I should have been doing is following the documentation for the version (3.1.10) that I had installed for my project, which is available directly from the associated npm page.

Smoother operations

What a difference a Promise makes! I was able to (at least syntactically) make my web routes work using the MongoClient.connect() Promise. I created a function for each CRUD operation that receives the necessary input arguments for the associated MongoDB collection method. Each CRUD function also receives a resCallback sent from within the HTTP method router which sends a response after MongoDB does its task: 


        const conn = MongoClient.connect(url, options);

        function connectAndCRUD(doc, callback) {
            conn.then( 
                // perform CRUD operation
                callback(message);
            )
        }

        //...later on in the file...//

        app.route('/path/to/route')
           .get(function(res, req) {

               function resCallback(obj) {
                   res.send(obj);
               }

                // do GET logic
                connectAndFind(somethingToFind, resCallback);

            });

Some of the intangible topics that came up which are not really in my immediate vision as I work through my projects:

  • Caching
  • Server response time

I think the core concept I need to learn, in order to better understand that episode and related content, is that of caches and memory usage in general. MongoDB speaks of this topic in their manual, and two main points stand out:

Technically, the operating system’s virtual memory subsystem manages MongoDB’s memory. This means that MongoDB will use as much free memory as it can, swapping to disk as needed. Deployments with enough memory to fit the application’s working data set in RAM will achieve the best performance.

MongoDB keeps most recently used data in RAM. If you have created indexes for your queries and your working data set fits in RAM, MongoDB serves all queries from memory. MongoDB does not cache the query results in order to return the cached results for identical queries. For more information on MongoDB and memory use, see WiredTiger and Memory Use and MMAPv1 and Memory Use.
