Project Journal: February 4, 2019

We interrupt this regularly scheduled program with continuous integration and deployment!

et130webapp

Two major (and majorly satisfying) tasks I completed today:

  • Setup Travis CI with the Github repository
  • Connect Github repository with Heroku and allow contiuous deployment if integration tests pass

Travis CI

I was having some issues with installing travis locally with rbenv/rvm but I was successfully able to do so with brew. After that, the major hurdle was setting up .travis.yml and then getting the tests to run successfully (and pass) upon each push to master.

After some googling, I created the yml file to be short and sweet: 


language: node_js
node_js:
    - "stable"

cache:
    directories:
    - "node_modules"

I am not sure if these are the "best" configuration values for what I will need in the future, but at this point, I wanted to do the bare minimum and get the app running on heroku.

My Mocha tests were not exiting/closing the server, and while there are options to close() the port connection, I instead chose to add --exit to my test script in package.json: 


"test": "mocha --exit --ui tdd ./src/test"

I also dumped my unit tests into the src/test folder so that travis would run them as well.

Heroku listens to its own port

Although my tests were passing and Github was sending off the pushes to Heroku, I kept getting a code=H10 desc="App crashed" message in the heroku logsfor all of the GET requests.

During the development phase, I had explicitly forced the server to listen on port 8080, and Heroku listens to its own port so this would not suffice. After a number of attempts, I decided to fallback on this approach I had seen used in quite a number of tutorials/SO answers: 


const port = process.env.PORT || 8080;

Since I had not declared a PORT in my .env file, and even if I had done so that file was being .gitignored anyway, the line above would allow both Travis and Heroku to do their thing.
